Most "Rust FFI" posts are about one direction: Rust → JS, Rust → Swift, Rust
→ Kotlin. Picking the binding tool is half the work, and the other half is
convincing your build system to play along.

Shipping an MPC wallet where one party is a phone and the other is a smart
card forces you into a less comfortable version of the problem: the crypto
core has to run on **both** targets, linked two different ways, from the
**same source tree**. The phone wants a Kotlin/Swift facade. The card wants a
hand-crafted C header and a `no_std` build with a custom allocator. Neither
tool is happy being told the other exists.

This is the story of what that costs.

## The two FFIs, at a glance

```text title="targets"
┌────────────┬────────────────┬──────────────┬────────────────────┐
│ target     │ env            │ binder       │ runtime assumption │
├────────────┼────────────────┼──────────────┼────────────────────┤
│ phone      │ Android / iOS  │ UniFFI       │ std, heap, async   │
│ active-card│ Cortex-M + RTX │ cbindgen + C │ no_std, no heap*   │
└────────────┴────────────────┴──────────────┴────────────────────┘
* we do have a heap via a custom allocator; std is still off.
```

The crate graph ends up shaped like this:

```text title="crate layout"
mpc-core              ← pure logic, no FFI, no std-only APIs
├─ mpc-sdk-kmp        ← UniFFI surface (phone)
└─ mpc_sdk_embedded   ← cbindgen surface (card), #![no_std] + alloc
```

Anything that flirts with `std::thread`, `std::fs`, `tokio`, `chrono`, or
`std::time::SystemTime` in `mpc-core` is a build-break on the card. This
constraint is great for code quality. It is hostile for half the crates on
crates.io.

## UniFFI on the phone is the easy side

The mobile binder is almost boring. You write a `.udl` or `#[uniffi::export]`
proc-macro surface, name the types, and Kotlin lands in your lap:

```rust title="mpc-sdk-kmp/src/lib.rs"
#[derive(uniffi::Error, thiserror::Error, Debug)]
pub enum MpcError {
    #[error("bad state: {0}")] BadState(String),
    #[error("ble timeout")]    BleTimeout,
    #[error("proto: {0}")]     Proto(String),
}

#[uniffi::export]
pub fn start_presign(group_id: String) -> Result<Vec<u8>, MpcError> {
    mpc_core::presign::start_round1(&group_id)
        .map_err(Into::into)
}
```

The generator emits a `MpcError` sealed class, a suspending helper for
callbacks, and Kotlin-idiomatic shapes. The friction is real but predictable:

- `Option<T>` becomes `T?` — fine, but nested `Option<Option<T>>` is an error
  the generator won't catch, it just flattens. Check the output.
- `Vec<u8>` becomes `ByteArray`. Fine for data; annoying for opaque handles
  because every call re-hashes into a Kotlin-owned buffer.
- Async callbacks require `uniffi::export(with_foreign)` and a matching
  `ForeignExecutor`. The docs are light; the test is whether exceptions
  thrown in Kotlin callbacks make it back across.
- Enums with associated data are supported, **but** if the generator can't
  see the type (private module, cfg-gated), you get a silent `Unknown`
  variant in Kotlin and a call that panics only at runtime.

What actually cost us time on mobile was none of the above. It was **build
integration**. KMP's Gradle plugin, the UniFFI Gradle wrapper, and the cargo
profiles per ABI don't compose until you accept that `publishToMavenLocal`
will fail on macOS targets because of linker quirks:

```sh title="mpc-sdk-rs/kmp-wrapper — works"
./gradlew publishAndroidReleasePublicationToMavenLocal \
  publishIosArm64PublicationToMavenLocal \
  publishIosSimulatorArm64PublicationToMavenLocal \
  publishKotlinMultiplatformPublicationToMavenLocal
```

```sh title="same, but broken on macOS — linker can't find _swiftinit"
./gradlew publishToMavenLocal
```

The fix is not a fix, it's a known scope: publish only the targets you
actually ship. Forty-five minutes lost the first time; instant skip every
time after.

## cbindgen on the card is where the mask comes off

The card runs on a ThreadX-based Cortex-M with ~1 MB of usable flash for the
MPC layer and no `std`. `mpc_sdk_embedded` is `#![no_std]` + `extern crate
alloc;` with a custom allocator that talks to ThreadX' byte-pool. The binder
is plain C:

```c title="public header (generated by cbindgen)"
typedef struct MpcSession MpcSession;
typedef struct MpcResult {
    int32_t  code;
    uint8_t* data;
    uintptr_t len;
} MpcResult;

MpcSession* mpc_session_new(const uint8_t* seed, uintptr_t len);
MpcResult   mpc_presign_begin(MpcSession*, const char* group_id);
void        mpc_result_free(MpcResult);
void        mpc_session_free(MpcSession*);
```

Every line of that header is a contract the firmware's C side has to honour
by hand. cbindgen will happily emit a header that lies if you let it.

Three lessons paid in blood:

**1. `Result<T, E>` is not a C type.** You have to flatten it into
`{code, data, len}` or similar and document who frees what. We forgot to
expose `mpc_result_free` for one week and leaked 200 bytes per presign until
a stress run found it:

```rust title="the flattening"
#[repr(C)]
pub struct MpcResult {
    pub code: i32,
    pub data: *mut u8,
    pub len: usize,
}

#[no_mangle]
pub extern "C" fn mpc_result_free(r: MpcResult) {
    if !r.data.is_null() && r.len > 0 {
        // SAFETY: r.data was produced by Box::into_raw on a Box<[u8]> of
        // length r.len in this same crate, on this same allocator.
        unsafe { drop(Box::from_raw(core::slice::from_raw_parts_mut(r.data, r.len))) }
    }
}
```

**2. `panic = "abort"` is the default, `panic_handler` is your problem.**
cargo-bloat doesn't help when your panic path pulls in `fmt::Write` and
quadruples your image size. We ended up with a panic handler that writes the
location to the log ring buffer and halts — no `format_args!` machinery in
the final binary.

```rust title="panic path, thin version"
#[panic_handler]
fn on_panic(info: &PanicInfo) -> ! {
    if let Some(loc) = info.location() {
        log_u32(loc.line());      // line number only — no fmt
        log_str(loc.file());      // &'static str, already in .rodata
    }
    loop { cortex_m::asm::wfi() }
}
```

**3. The allocator bridge is where integer overflow lives.** `alloc::vec`
calls through the `GlobalAlloc` trait, which hands you a `Layout`. Forgetting
to check that `layout.size() == 0` against ThreadX' byte-pool returning
`NULL` for zero-byte allocations cost a Saturday.

## The shared core pays the price

`mpc-core` has to be clean enough that neither binder has to apologise for
it. That means:

- `#![no_std]` compatible — no `println!`, no `std::collections::HashMap`
  (use `hashbrown` or `BTreeMap`).
- No `thread_local!`, no `Mutex<T>` as default sync — mobile wraps to
  `parking_lot::Mutex`, card wraps to a ThreadX mutex.
- No hidden global state. Every session carries its own context. This makes
  the FFI layers much smaller: the binder is a thin handle factory, not a
  lifecycle manager.
- Avoid generics on public API. UniFFI can't generate bindings for
  generics, and cbindgen will silently emit a monomorphised header that
  lies about the ABI.

The payoff: `mpc-core` has its own test suite running on host `cargo test`,
and neither FFI layer ever has to retest cryptographic correctness — only
the marshalling.

## What the build system looks like at rest

```sh title="one command, both targets"
# card
cd daric_sdk/rust/mpc_sdk_embedded
cargo build --release --target thumbv7em-none-eabihf

# phone
cd ../mpc-sdk-rs/kmp-wrapper
./gradlew publishAndroidReleasePublicationToMavenLocal \
  publishIosArm64PublicationToMavenLocal \
  publishIosSimulatorArm64PublicationToMavenLocal \
  publishKotlinMultiplatformPublicationToMavenLocal
```

Two invocations, same `mpc-core`, two very different ABIs out the other end.
The linker error rate dropped to near zero the day we stopped trying to
build them in the same Gradle pass.

## When you need two FFIs

You don't, usually. If your target is "native app + desktop CLI", UniFFI or
Neon will carry you a long way. You need this split when one of your
endpoints is a hardware device with its own firmware and its own supply
chain — and when the cost of bugs diverging between a phone SDK and a
firmware layer is higher than the cost of owning a shared crate.

Which is exactly the shape of a 2-party MPC wallet. So here we are.
